# 가상 시나리오 분포 정책 (초안 v1)

## 1. 목적과 원칙

- 대상은 실제 방문 기록이 아니라 **가상 시나리오 기반 인간 판단 데이터**다.
- 따라서 시나리오는 실제 이용자 분포를 흉내 내되, 희소하지만 추천을 크게 바꾸는 조건도 충분히 포함해야 한다.
- 모든 시나리오에 하나의 고정 비율을 적용하지 않는다. 새 run을 만들 때마다 허용 범위 안에서 가중치를 난수로 추출한다.
- run이 시작되면 `seed`, 허용 범위, 실제 추출 가중치, 실제 건수를 함께 기록하고 변경하지 않는다.
- 결측은 `UNKNOWN`이다. 특히 주차 정보가 없는 점포를 "주차 불가"로 처리하지 않는다.

## 2. 현실 분포와 학습 분포의 혼합

시나리오 분포는 아래 두 층을 섞는다.

| 층 | 권장 비중 | 역할 |
|---|---:|---|
| 현실성 층 | 70~80% | 시간대·활동·이동 조건이 실제 한남권 이용 맥락에서 크게 벗어나지 않도록 한다. |
| 학습 보강 층 | 20~30% | 주차 필수, 1시간 이하, 낮은 예산처럼 드물지만 추천 결과를 바꾸는 조건을 의도적으로 포함한다. |

이는 실제 이용자 비율을 그대로 복제하는 정책이 아니다. 희소 조건을 너무 적게 만들면 모델이 해당 조건을 학습하거나 검증할 수 없기 때문이다.

## 3. run 단위 난수 분포 생성 규칙

1. 각 조건의 가중치를 아래 범위 안에서 난수 추출한다.
2. 같은 필드의 가중치 합이 100%가 되도록 정규화한다.
3. 240건 기준 최소·최대 건수 범위를 검사한다.
4. 범위를 벗어나거나 객관적 후보가 없는 조합은 다시 추출한다.
5. 확정된 가중치로 240건을 생성한다.

가중치는 **시나리오 1건마다 다시 난수로 바꾸지 않는다.** 한 run 안에서는 하나의 확정된 분포를 사용해야 재현과 분석이 가능하다.

## 4. v1 기본 범위

아래 숫자는 현재 공개 통계와 마스터 데이터만으로 확정할 수 없는 항목에 대한 **운영 시작값**이다. 실제 서비스 초기 화면의 선택 로그와 한남권 상권 데이터를 확보하면 v2에서 재보정한다.

### 4.1 강한 이동 제약

| 필드 | 코드 | run별 허용 비율 | 240건 환산 | 근거와 주의점 |
|---|---|---:|---:|---|
| 주차 조건 | `REQUIRED` | 10~20% | 24~48건 | 서울 전체 승용차 통행 비중을 그대로 쓰지 않는다. 한남권 다음 활동 선택이라는 더 좁은 맥락이며, 희소 조건 학습을 위해 최소량을 둔다. |
| 주차 조건 | `NO_PREFERENCE` | 나머지 | 192~216건 | `parking_available=UNKNOWN` 점포는 주차 불가로 제외하지 않는다. |
| 경사 | `AVOID` | 15~35% | 36~84건 | 한남권 지형을 고려한 보강 조건이다. 실제 사용자 로그가 생기면 조정한다. |
| 경사 | `NO_PREFERENCE` | 나머지 | 156~204건 |  |

### 4.2 분위기

분위기는 현재 우선 현실성보다 다양성을 목적으로 한다. 세 코드의 비율을 각각 25~40% 범위에서 추출하되, 합계가 100%가 되도록 한다.

| 코드 | 화면 표현 |
|---|---|
| `QUIET` | 조용함 |
| `LIVELY` | 활기참 |
| `NO_PREFERENCE` | 분위기 상관없음 |

`편안함`은 마스터 정규화 규칙에 따라 `QUIET`으로 사용한다.

### 4.3 시간과 요일

시간대는 생활인구·상권 시간대 매출 데이터로 보정할 대상이다. 데이터 결합 전에는 아래 구간별 범위를 사용하고, 구간 안의 정각은 균등 추출한다.

| 구간 | 방문 시각 | run별 허용 비율 |
|---|---|---:|
| 오전 | 09:00~11:00 | 10~20% |
| 점심 | 12:00~14:00 | 20~35% |
| 오후 | 15:00~17:00 | 20~35% |
| 저녁 | 18:00~20:00 | 25~40% |

- 평일 전체와 주말 전체의 비율도 각각 50~65%, 35~50% 범위에서 추출한다.
- 이후 서울 생활인구와 한남권 상권의 시간대별 소비 데이터를 결합해 v2로 갱신한다.

### 4.4 동행과 인물 정보

초기 화면의 조건부 입력 구조를 그대로 지킨다.

| 동행 | 연령대 | 성별 |
|---|---|---|
| `FAMILY` | 생성하지 않음 | 생성하지 않음 |
| `COUPLE` | 생성 | 생성하지 않음 |
| `FRIENDS` | 생성 | 생성하지 않음 |
| `ALONE` | 생성 | 생성 |

동행·연령·성별의 실제 한남권 방문 비율은 현재 근거 데이터가 없다. v1에서는 동행 유형을 15~35% 범위 안에서 난수 추출하고, 연령대는 해당 동행 안에서 균등 추출한다. 실제 초기 화면 선택 로그가 확보되면 이를 우선 근거로 교체한다.

### 4.5 남은 시간과 다음 활동

| 필드 | 코드 | v1 run별 허용 비율 |
|---|---|---:|
| 남은 시간 | `UNDER_60` (60분 이하) | 15~30% |
| 남은 시간 | `FROM_60_TO_120` | 25~40% |
| 남은 시간 | `OVER_120` | 10~25% |
| 남은 시간 | `NO_PREFERENCE` | 20~35% |
| 다음 활동 | `EAT`, `REST`, `BROWSE`, `EXPERIENCE`, `NO_PREFERENCE` | 각 10~30%, 합계 100% |

활동별 실제 비율은 상권의 업종별·시간대별 소비 통계와 서비스 사용자 로그를 결합해 보정한다.

### 4.6 예산: 활동별 조건부 생성

예산은 모든 활동에 같은 분포를 적용하지 않는다. `FREE`는 구매가 필수가 아닌 구경·일부 문화 활동에서는 자연스럽지만, 식사에서는 기본적으로 생성하지 않는다.

| 다음 활동 | `FREE` | `UNDER_10000` | `UNDER_30000` | `UNDER_50000` | `NO_PREFERENCE` |
|---|---:|---:|---:|---:|---:|
| `EAT` (식사) | 0% | 10~25% | 40~60% | 10~25% | 나머지 |
| `REST` (쉬기) | 0~5% | 25~45% | 25~45% | 5~15% | 나머지 |
| `BROWSE` (구경) | 35~60% | 5~20% | 5~20% | 0~10% | 나머지 |
| `EXPERIENCE` (새로운 경험) | 10~30% | 10~25% | 25~45% | 5~20% | 나머지 |
| `NO_PREFERENCE` | 0~20% | 10~30% | 20~40% | 5~20% | 나머지 |

각 행은 범위 안에서 앞의 네 비율을 추출한 뒤, 합계가 100% 이하일 때만 나머지를 `NO_PREFERENCE`로 둔다. 모든 비율은 5% 단위로 반올림해 해석 가능한 run 프로필을 만든다.

## 5. 현재 데이터의 제약

- MVP 30개 점포 모두 `parking_available=UNKNOWN`이다. 주차 필수 시나리오의 비율은 만들 수 있으나 주차 가능성에 따른 추천 품질은 검증할 수 없다.
- MVP의 F&B 13개 중 대표 가격이 알려진 곳은 12개이며, 이는 대표 메뉴 가격이지 1인 실제 지출액이 아니다.
- 그러므로 위 예산 범위는 점포 가격의 실측 분포가 아니라 가상 시나리오의 시작 정책이다.
- `FREE` 예산의 하드 필터는 알려진 유료 F&B에만 적용한다. 소매·문화·뷰티는 구매하지 않고 둘러볼 수 있으므로 가격이 있어도 자동 제외하지 않는다.

## 6. 근거 데이터와 v2 갱신 기준

| 갱신 대상 | 우선 근거 | 갱신 시점 |
|---|---|---|
| 요일·시간대 | 서울 생활인구, 한남권 상권 시간대 매출 | 공개 데이터 결합 후 |
| 활동·연령·성별 | 서비스 초기 화면 선택 로그 | 충분한 실제 입력이 누적된 뒤 |
| 예산 | 한남권 업종별 카드 소비와 점포별 대표 가격·가격대 | 가격 결측 보완 후 |
| 주차 | 사용자 입력 로그와 점포별 주차 실측값 | 주차 데이터 보완 후 |

참고: 서울시 상권분석서비스는 상권 단위의 추정매출·시간대별 매출 건수·연령/성별 매출 데이터를 제공한다. 서울 생활인구는 지역·시점별 체류 인구 보정에 사용한다.

## 7. run manifest에 반드시 저장할 값

```json
{
  "distribution_policy_version": "v1",
  "seed": 43,
  "configured_ranges": {},
  "drawn_weights": {},
  "generated_counts": {},
  "data_limitations": ["parking_available is UNKNOWN for all MVP stores"]
}
```

## 8. 변경 절차

1. 이 문서의 범위 또는 근거를 바꾸면 버전을 올린다. 예: `v1` → `v2`.
2. 이미 시작한 run의 분포·seed·스냅샷은 바꾸지 않는다.
3. 새 정책은 새 run에서만 적용한다.
4. 기존 인간 라벨과 자동·의사 라벨은 출처를 분리해 저장·평가한다.
